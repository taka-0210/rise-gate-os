# 共通スタッフ基盤 Phase 1 設計

## 目的

ライズアップの各業務システムが、同じスタッフ情報・所属・在籍状態・利用権限を安全に参照できる共通基盤を作る。

この基盤は認証と業務上の本人識別を担当する。給与計算や労務管理で使う詳細な人事情報は、原則として共通認証データから分離する。

## 情報の分離方針

### 共通スタッフ情報

各システムが安全に共有できる最小限の情報。

| 項目 | 用途 | 必須 | 備考 |
|---|---|---:|---|
| staff_id | システム内部の不変ID | 必須 | ULID。画面上の社員番号とは分離する |
| employee_code | 既存名簿・給与システムとの照合 | 必須 | 変更履歴を残す。ログインIDには使わない |
| display_name | 画面・履歴への氏名表示 | 必須 | 氏名変更に対応する |
| name_kana | 検索・並び替え | 任意 | 正規化した検索値を別途持てる |
| login_id | 共通ログイン | 必須 | 一意。既存値の重複・空欄を取込前に検査する |
| employment_type | 権限判断の補助 | 任意 | 正社員・役員・パート等。自由入力ではなくマスター化する |
| employment_status | ログイン可否と在籍状態 | 必須 | active / leave / retired |
| joined_on | 在籍期間の判定 | 任意 | 認証には直接使用しない |
| left_on | 退職後の利用停止 | 任意 | 退職しても過去の履歴は削除しない |
| login_enabled | 緊急停止を含むログイン可否 | 必須 | 在籍状態と分けて即時停止できる |

### 制限付き人事情報

次の情報は、共通認証の必須項目にしない。必要な場合は人事情報領域へ保存し、総務など許可された役割だけが閲覧できるようにする。

- 生年月日
- 性別
- 現住所
- 雇入・退職の詳細理由
- 契約・給与・社会保険に関する情報

パスワード再発行時の本人確認にも、生年月日や住所をそのまま秘密情報として使用しない。

## 所属構造

スタッフと所属先は多対多で管理する。主所属を1件持ち、応援・兼務・異動に対応する。

### organizations

会社を表す。初期段階では株式会社ライズアップを登録する。

### workplaces

店舗・本社・再生センターなど、業務上の所属単位を表す。

### departments

営業部・総務部・商品管理部などの部門を表す。店舗とは別軸で管理する。

### staff_assignments

| 項目 | 内容 |
|---|---|
| staff_id | 対象スタッフ |
| workplace_id | 所属店舗・拠点 |
| department_id | 所属部門。未設定を許可 |
| is_primary | 主所属かどうか |
| started_on | 所属開始日 |
| ended_on | 所属終了日 |

同一期間の主所属は原則1件に制限する。過去の所属は削除せず終了日を設定する。

## システム利用権限

所属とシステム権限を分離する。特定店舗に所属していることだけを理由に、すべてのシステムへ入れる設計にはしない。

### systems

環境整備管理、hit_portalなどの接続先システムを登録する。

### system_roles

システムごとの役割を登録する。例：管理者、一般、閲覧のみ。

### staff_system_access

スタッフ単位で利用可否と役割を付与する。有効期間と停止日時を持ち、退職・異動・緊急停止に対応する。

## 在籍・異動時のルール

1. 入社時はスタッフを登録し、主所属と必要なシステム権限を別々に付与する。
2. 異動時は既存所属へ終了日を設定し、新しい所属を追加する。
3. 休職時は在籍情報を残し、業務上必要なシステム権限だけを停止する。
4. 退職時はログインを即時停止し、所属・権限へ終了日を設定する。
5. スタッフを物理削除しない。投稿・完了・承認など過去の業務履歴は staff_id で保持する。
6. 氏名と所属を帳票や操作履歴へ表示する場合は、必要に応じて記録時点の表示名をスナップショット保存する。

## 名簿CSVの取込方針

- 原本は非公開領域で取り扱い、Gitへ保存しない。
- プレビュー、検証、確定の3段階に分ける。
- 従業員コードを既存データとの照合キー候補にする。
- ログインIDの空欄・重複をエラーとして表示し、人が確定するまで登録しない。
- 所属文字列は、店舗・部署マスターへの変換結果を確認してから登録する。
- 住所・生年月日など、共通基盤に不要な列は初期取込対象から除外する。
- 誰がいつどのファイルを取り込み、何件追加・更新・除外したかを監査ログへ残す。

## Phase 2へ進むための決定事項

- `staff_id` はULID、`employee_code` と `login_id` は別項目とする。
- 複数所属を許可し、主所属を1件設定する。
- 退職・休職・異動でスタッフ本体を削除しない。
- 認証情報、共通プロフィール、制限付き人事情報を分離する。
- システム利用権限はスタッフ単位・システム単位で管理する。
- 名簿CSVの個人情報をソースコードや提案ログへ転記しない。

## 次の実装単位

新しい専用アプリケーション `hit_staff` を作成し、最初に次を実装する。

1. staffs、workplaces、departments、staff_assignments のテーブル
2. systems、system_roles、staff_system_access のテーブル
3. スタッフ一覧・登録・編集・利用停止
4. CSVの検証プレビュー
5. 認証情報の安全な登録とパスワード変更
