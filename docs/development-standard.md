# RISE GATE 開発・デプロイ標準

## 目的

この文書は、RISE GATE が開発するすべての案件で、Git運用とデプロイ運用を統一するための標準を定義します。

案件ごとの認識差による、意図しないPush、デモ更新、本番反映を防ぎます。

共通ルールは個人Codexスキル `rise-gate-development` に置き、案件固有ルールは各リポジトリ直下の `AGENTS.md` に置きます。

## 基本原則

```txt
Codex
  実装
    ↓
  検証
    ↓
  関連ファイルだけコミット
    ↓
  GitHubへPush

ユーザー
  GitHub Actionsを手動実行
    ↓
  デモまたは本番へデプロイ
```

- GitHubへのPushとサーバーへのデプロイは別の操作です。
- Pushを契機にデモや本番へ自動デプロイしません。
- デプロイワークフローは `workflow_dispatch` のみ使用します。
- Codexは検証後のコミットとPushまで担当します。
- ユーザーがGitHub Actionsの `Run workflow` からデプロイします。
- 認証情報はGitHub Actions Secretsに保存し、コード、ログ、チャットへ出しません。
- サーバー上のDB、アップロード、問い合わせ、セッション、環境設定、生成データをデプロイで失わないようにします。

## 新規案件の開始

### 1. Projectを作る

Rise Gate OSにProjectを作成します。

決めること:

- Project名
- Client / Company
- 目的と概要
- 担当者
- 開始日と期限
- デモ確認日
- 本番公開予定

### 2. Improvementを登録する

新規開発も、まずProject内のImprovementから始めます。

最低限、次を残します。

- 現状
- 問題
- 期待する状態
- 今回の実装範囲
- 今回やらないこと
- 次の行動

### 3. Gitリポジトリを準備する

- ローカルの案件フォルダを作成します。
- GitHubに案件リポジトリを作成します。
- `.gitignore` を設定します。
- APIキー、パスワード、顧客データ、セッション、実問い合わせをコミット対象から外します。
- GitHub Pagesは、静的サイトとして公開する案件以外では無効にします。

### 4. `AGENTS.md` を配置する

各リポジトリ直下へ、RISE GATE共通テンプレートを元に `AGENTS.md` を配置します。

案件固有情報として次を追記します。

- 使用技術
- 検証コマンド
- ブランチ名
- デモURL
- 本番URL
- デプロイワークフロー
- サーバー上の保護対象

## 通常の開発フロー

```txt
ユーザーがCodexへ依頼
  ↓
Codexが作業範囲と既存変更を確認
  ↓
実装
  ↓
関連テストと構文検査
  ↓
関連ファイルだけコミット
  ↓
GitHubへPush
```

検証失敗、認証切れ、競合、変更範囲不明がある場合、Codexは強制的にPushせず、阻害要因を報告します。

## デプロイワークフロー

ワークフロー名を次に統一します。

```txt
デモへデプロイ
本番へデプロイ
```

起動条件:

```yaml
on:
  workflow_dispatch:
```

案件構成に応じて内部処理を選びます。

- 軽量PHP: FTPS差分同期
- Laravel: 自動テスト、成果物作成、SSH転送、リリース切替
- 静的サイト: ビルド成果物の配置

操作は統一し、内部実装だけを案件に合わせます。

## デモ確認まで

デモ確認日までは、必要なタイミングでユーザーが `デモへデプロイ` を実行します。

```txt
GitHub
  Actions
    ↓
  デモへデプロイ
    ↓
  Run workflow
```

デプロイ後は対象URLを開き、主要画面、フォーム、管理画面、保存処理を確認します。

## クライアント確認後

クライアントが確認したデモは固定します。

- Codexはローカルでブラッシュアップします。
- コミットとPushは継続します。
- Pushだけではデモは変化しません。
- 次に見せる準備が整った時だけ、ユーザーが手動デプロイします。

これにより、作業途中の変化を見せず、まとまった完成版を提示できます。

## 本番公開

- デモで承認されたコミットを特定します。
- 関連テストを通します。
- バックアップと切り戻し方法を確認します。
- ユーザーが `本番へデプロイ` を手動実行します。
- 公開URLと主要機能を確認します。
- サーバー所有データが保持されていることを確認します。

## 状態の伝え方

作業報告では次の表現を使い分けます。

- `GitHubへPush済み`: ソース履歴へ反映済み。サーバーは変化していません。
- `デプロイ準備済み`: 手動ワークフローを実行できます。サーバーは変化していません。
- `デプロイ完了`: ワークフローが成功し、対象URLも確認済みです。

## 新規案件でCodexへ伝える言葉

```txt
RISE GATEの新規案件として開始する。
rise-gate-development標準でセットアップして。
```

Codexは個人スキルとリポジトリの `AGENTS.md` を組み合わせ、共通運用と案件固有ルールの両方を適用します。
